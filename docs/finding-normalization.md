# Finding Normalization

Security scanners produce different output formats. The analyzer needs a common schema before it can compare, correlate, or score findings from multiple tools.

The first normalization step supports Semgrep JSON output.

## Input

Semgrep writes raw scan output to:

```text
scanner-results/semgrep.json
```

This file is generated by the CI workflow or by a local Semgrep run.

## Output

The analyzer writes normalized findings to:

```text
scanner-results/normalized-findings.json
```

The output file is generated runtime data and is not committed to Git.

## Common Finding Fields

The initial normalized schema keeps the fields needed for risk analysis:

| Field | Purpose |
| --- | --- |
| `scanner` | Source tool, such as `semgrep`. |
| `rule_id` | Scanner rule or check identifier. |
| `title` | Human-readable finding message. |
| `severity` | Normalized severity such as `low`, `medium`, or `high`. |
| `category` | Security domain, initially `sast`. |
| `file` | File where the finding was detected. |
| `line` | Starting line of the finding. |
| `end_line` | Ending line, if available. |
| `cwe` | Related CWE identifiers. |
| `owasp` | Related OWASP category mappings. |
| `confidence` | Scanner confidence signal. |
| `impact` | Estimated impact signal. |
| `likelihood` | Estimated likelihood signal. |

## Run

From the repository root:

```powershell
.\.venv\Scripts\python.exe -m analyzer.normalize_semgrep
```

Expected output:

```text
Normalized 1 Semgrep finding(s) to scanner-results/normalized-findings.json
```

## Why This Matters

Raw scanner output is useful, but each tool speaks its own format. Normalization creates a stable internal representation that future risk scoring can use regardless of whether the finding came from Semgrep, Gitleaks, Trivy, Checkov, or another scanner.
